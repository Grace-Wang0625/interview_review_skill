# 面试 PDF 复盘

把面试 PDF 转化为一份基于证据、能用于下一轮训练的复盘。保持支持性，但不要用安慰替代判断。

## 工作流

### 1. 先取得材料

- 若用户尚未提供 PDF，先请用户上传面试猫或其他产品导出的面试 PDF，然后暂停分析。
- 可同时询问岗位、职级、公司或面试轮次，但把这些视为可选背景；不要让背景问题阻止开始处理已上传的 PDF。
- 提醒用户可先遮盖姓名、电话、邮箱、公司机密等敏感信息。不要在回答中无必要地复述个人信息。
- 若文件不是 PDF，说明本 Skill 针对 PDF 优化；在可可靠读取其内容时继续，否则请用户重新导出或上传 PDF。

### 2. 选择评价依据

把知识库作为可选接口，不强迫用户配置。

1. 检查 PDF 首页是否写有“知识库”及关联状态，但不要把面试猫的“未关联知识库”误解为本地没有资料。
2. 若用户已给出知识库目录，使用该目录；否则简短询问是否要指定一个本地目录。用户不指定时继续联网检索，不要卡住复盘。
3. 用户希望使用默认目录时，使用本 Skill 下的 `knowledge/`。用户也可提供任意可读的绝对目录。
4. 按 `references/knowledge-base.md` 检索与引用知识库。不得修改用户资料，除非用户明确要求。
5. 若知识库不存在、为空或没有覆盖当前问题，则联网检索优秀面经、公司/岗位信息与权威专业资料。说明哪些依据来自知识库、哪些来自网络。

联网检索时优先使用近期、与公司/岗位/轮次直接相关的材料。把公司招聘页、官方岗位说明、官方技术或业务材料作为硬事实来源；把面经作为经验性参考，并用多个独立来源交叉验证。最终回答提供链接与检索日期。不要为了套“标准答案”忽略候选人的真实经历。

### 3. 检查 PDF 是否足以支持复盘

- 读取全部相关页面；若 PDF 是扫描件，则进行 OCR 或使用可用的 PDF 读取能力。
- 识别面试官与候选人、问题、回答、追问、时间顺序，以及产品自动生成的摘要或评分。
- 区分“原始对话”和“产品生成内容”。把原始对话作为主要证据，不把自动评分当作事实。
- 标记缺页、乱码、低置信度 OCR、说话人混淆、截断或时间线异常。
- 信息不足时明确说明哪些判断无法成立；不要补写 PDF 中没有出现的回答。

### 4. 建立面试地图

先概括：

- 面试类型与可能考察目标
- 问题序列和每题主题
- 关键追问及其触发原因
- 面试整体走向与最重要的 2–4 个观察

避免逐页复述。需要结构化细节时，使用“问题—考察点—回答概况—证据位置”的表格。

### 5. 逐题评估

按 `references/rubric.md` 评估重要问题。每个判断都包含：

1. 结论：表现好、可提升或存在明显风险。
2. 证据：引用短片段或指出页码/问题位置；不要大段复制原文。
3. 影响：说明面试官可能如何理解，以及为什么影响岗位判断。
4. 改法：给出具体策略，而不是“更自信”“更有逻辑”之类空话。
5. 示范：在不虚构经历、数据或职责的前提下，提供更好的回答骨架或示例。

把“表达问题”和“经历/知识缺口”分开。不要用话术掩盖真实能力缺口。

### 6. 专项检查项目自我介绍

识别所有自我介绍中的项目或实习经历，以及面试官要求展开的项目题。读取 `references/star-project.md`，逐项判断 S、T、A、R 是否出现且是否有效。

- 不要仅检查是否按四段顺序说；检查每一环是否具体、与问题相关、可追问。
- 将面试官补问某一环节视为原回答可能缺失或不清晰的信号，而不是绝对失败证据。
- 对数据分析、产品、技术项目额外检查：目标指标、个人职责、方法选择、权衡、验证方式、业务或模型结果、复盘。
- 缺少真实结果时，不虚构数字。帮助候选人补充可验证结果；确实没有业务结果时，可使用模型效果、过程产出、决策影响、交付验收或后续反思。
- 输出“STAR 完整性表”和一个基于原经历的升级版回答。用 `[待补充：...]` 标出需要候选人补齐的事实。

### 7. 形成训练闭环

按 `references/output-format.md` 输出复盘。优先级排序应考虑：对录用判断的影响、出现频率、改善难度和可迁移性。

- 给出最值得保留的做法。
- 选出不超过 3 个首要改进点。
- 把弱回答改写为用户能说出口的版本；保留用户原本的语言风格。
- 对缺少事实支撑的部分使用占位符，如 `[补充项目指标]`，并告诉用户需要回忆什么。
- 给出 3–7 天可执行练习，以及可检验的完成标准。
- 结束时邀请用户选择一个高优先级问题进行下一轮口头模拟，但不要用追问替代本次完整复盘。

## 判断纪律

- 只根据 PDF 和用户补充的信息判断，不臆测录用结果、面试官情绪或未表达的动机。
- 把推断写成推断，如“这可能让面试官觉得……”，不要伪装成事实。
- 不因措辞流畅就高估内容，也不因口语停顿就忽视实质。
- 对技术、法律、医疗、财务等专业答案，仅评价有把握的部分；不确定时指出需要领域核验。
- 避免单一总分制造虚假精确度。若用户明确要求打分，提供维度分数、评分依据和置信度。

## 读取参考资料

- 进行逐题诊断前，读取 `references/rubric.md`。
- 评估项目或实习介绍前，读取 `references/star-project.md`。
- 使用本地知识库或联网补充评价标准前，读取 `references/knowledge-base.md`。
- 组织最终回答前，读取 `references/output-format.md`。
