# 项目介绍的 STAR 检查

## 判断原则

STAR 是信息完整性工具，不是必须僵硬背诵的句式。允许候选人调整顺序，但听众应能清楚还原以下四部分。

| 环节 | 有效信息 | 常见缺口 |
|---|---|---|
| S 情境 | 业务/课程背景、问题、对象、约束、为什么重要 | 只说“老师给了数据集”；背景过长但没有问题 |
| T 任务 | 明确目标、成功标准、个人角色和责任边界 | 把团队任务当个人任务；只说“做预测”而无标准 |
| A 行动 | 候选人的分析、选择、步骤、权衡、协作和验证 | 罗列工具；大量使用“我们”而无法识别个人贡献 |
| R 结果 | 量化效果、业务/决策影响、交付结果和复盘 | 只说某模型最好；没有指标、基线、意义或落地 |

## 项目题的扩展检查

对数据分析、数据科学、产品和技术项目，在 STAR 之外检查：

- 数据来源、口径、质量与限制是否清楚。
- 方法是否服务于目标，是否解释为什么选择或放弃某方案。
- 评估指标是否与业务代价一致；是否提供基线、验证集或对照。
- 结果是否说清具体数值、相对提升、可解释性与实际用途。
- 候选人能否承认课程项目与真实业务之间的边界。

## 输出 STAR 完整性表

使用“充分 / 部分 / 缺失 / 无法判断”，不要制造虚假精确分数。每格写短证据与改善建议。

## 改写方法

1. 先用一句话交代问题和目标。
2. 用一句话说清个人职责与成功标准。
3. 选择 2–3 个最能体现判断力的行动，不把算法名当作主线。
4. 报告结果、与基线的比较及其意义。
5. 用一句复盘说明局限或下一步。

将升级稿控制在约 60–120 秒可口述长度。只使用 PDF 或用户确认过的事实；其余内容用 `[待补充：具体指标]` 等占位。
